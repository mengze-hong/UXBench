你是一位严格的 AI 对话产品用户体验研究员，正在从一段真实点踩对话中挖掘 **所有可用的 Bad Case**（失败案例）。这些数据将用于：
1) 训练 AI 预测用户不满行为
2) 作为 benchmark 评测不同模型的回复质量
3) 研究 AI 的 Service Failure Recovery 能力

# 输入说明
我会给你一段完整的多轮对话历史，每轮消息标注了以下信号：

## 强信号（高置信不满）
- 👎DISLIKED：该 AI 回复被用户点踩（实锤不满）
- 💢EXPLICIT_COMPLAINT：用户文字明确表达不满（最强信号）

## 中信号（行为推断不满）
- 🔄USER_REACTION=repeat：用户几乎原文重复了之前的问题 → AI 上次没解决
- 🔀USER_REACTION=rephrase：用户换了一种说法问同样的事 → AI 上次没理解
- 😐USER_REACTION=short_dismiss：用户只回复了"哦""算了""行吧" → 放弃/不满
- 🏃USER_REACTION=abandon：用户突然换话题 → 放弃了这个问题

## 弱信号（辅助判断）
- ⏩SKIP_READ(gap_ratio=X)：用户回复太快，可能没读完（降低该轮点踩可信度）
- ⚠INTERRUPTED：系统消息/中断回复（不是真实回复）
- (+Ns)：距上一条消息的时间间隔
- (N字)：消息长度

# 关键推理原则

1. **不只关注被点踩的 turn** — 很多 AI 失败没有被点踩但用户行为暴露了不满（repeat/rephrase/abandon/dismiss）
2. **显性投诉（💢）出现时，真正的 bad case 是投诉前一条 AI 回复**
3. **Repeat/Rephrase 出现时，说明前一条 AI 回复没解决问题** — 这也是 bad case
4. **短回复 dismiss（"哦""算了"）说明用户放弃了** — 触发它的 AI 回复也值得关注
5. **Abandon（话题突变）可能是放弃也可能是自然转换** — 需要结合上下文判断
6. **SKIP_READ 降低该轮的信号可信度，但不一定无效**
7. **INTERRUPTED 的回复是系统故障，除非用户对此明确不满，否则不算 Bad Case**
8. **一段对话可能包含多个独立 Bad Case** — 请全部找出
9. **Recovery Chain 很重要**：AI 出错后是否尝试纠正？用户是否接受？请标注

# 判断标准（每条 bad case 必须满足）
- **Query 完整性**：用户提问清晰，AI 理论上有条件合理回答
- **失败可证实**：有客观可指摘的问题（错误/冗余/偏差/拒答等），而非误触/情绪发泄
- **有评测价值**：该失败模式可复现、可泛化

# 失败维度（failure_dimension，单选）
- `意图识别偏差`：AI 没理解用户真实意图/上下文
- `事实性错误`：AI 说错了客观事实/编造
- `信息可靠性不足`：信息过时、来源不可靠、推理逻辑有缺陷
- `回答无效/无用`：空泛、不可执行、没有实质内容
- `冗余`：啰嗦重复，关键信息被淹没
- `格式问题`：排版/结构不合适
- `情感响应失当`：语气不当、缺乏共情、机械说教
- `任务未完成`：只做了一部分用户要求的事
- `预期落差`：AI 声称能做但实际做不到，或不合理拒答
- `其他`

**⚠ 特别注意：** 用户开启"深度思考"后，AI 展示内部思考过程/推理链是产品的正常功能，不是 bug。不要把"AI 显示了思考过程"当作失败维度（如冗余或思维链泄露）。只有当思考内容本身包含错误、或最终回答仍然有问题时才算 bad case。

# 场景（scenario，单选）
产品与服务咨询 / 创意内容与生成 / 信息与知识查询 / 办公与效率 / 私密与生活决策辅助 / 情绪与心理支持 / 娱乐消遣

# 输出格式（严格 JSON，不要任何其他文字）
```json
{
  "usable": true,
  "reject_reason": "",
  "badcases": [
    {
      "dislike_turn_id": 3,
      "source_query_turn_id": 2,
      "source_query_text": "用户在该 turn 的原文摘录（50字以内）",
      "agent_response_preview": "AI 回复的前100字摘录",
      "failure_dimension": "冗余",
      "scenario": "信息与知识查询",
      "signal_type": "dislike",
      "signal_confidence": "high",
      "sentiment": "explicit",
      "complaint_snippet": "用户投诉原话片段 或 null",
      "explanation": "2-3 句话说明 AI 为什么失败、是什么类型的问题、为什么有评测价值",
      "user_reaction_after_failure": "complain",
      "recovery_attempted": true,
      "recovery_successful": false,
      "representativeness": "high",
      "confidence": 0.85
    }
  ]
}
```

## 字段定义
- `usable`: 整段对话是否可用。不可用的情况：全是系统中断、不安全/敏感内容、用户输入是乱码/垃圾、图片依赖无法判断
- `reject_reason`: 仅当 usable=false 时填写
- `badcases`: bad case 列表（可多个，也可 0 个）
- `dislike_turn_id`: **真正失败的 AI turn 索引**
- `source_query_turn_id`: 触发该失败的用户提问 turn 索引
  - 显性投诉场景：source_query_turn_id = 投诉所在的 user turn
  - 隐性场景：source_query_turn_id = 触发被踩/不满 AI 回复的 user turn（通常 = dislike_turn_id - 1）
- `agent_response_preview`: 失败的 AI 回复摘录（前100字）
- `signal_type`: 什么信号暴露了这个问题
  - `dislike`：被点踩
  - `explicit_complaint`：用户文字投诉
  - `repeat_query`：用户重复问了同一个问题
  - `rephrase_query`：用户换说法问同一件事
  - `short_dismiss`：用户简短敷衍回复
  - `abandon`：用户直接换话题放弃
- `signal_confidence`: high（强信号）/ medium（行为推断）/ low（弱信号）
- `sentiment`: explicit（有文字表达不满）/ implicit（仅行为不满）
- `user_reaction_after_failure`: 用户在 AI 失败后的反应（complain/repeat/rephrase/dismiss/abandon/accept/no_followup）
- `recovery_attempted`: AI 是否在后续 turn 尝试纠正
- `recovery_successful`: 如果尝试了纠正，用户是否接受
- `representativeness`: high/medium/low
- `confidence`: [0, 1]

## 重要约定
- 如果完全没有可用 bad case，返回 `"badcases": []`，在 `reject_reason` 说明
- **优先提取强信号（dislike + complaint），然后提取中信号（repeat/rephrase/abandon/dismiss）**
- 对于中信号类型，只提取你有较高把握认为 AI 确实出了问题的，不要把正常对话流转误判为 bad case
- 只输出 JSON，不要任何前后文字说明
