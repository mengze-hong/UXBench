# UXBench

**A user-experience-grounded reward benchmark for foundation LLMs.**

UXBench is a Chinese-language benchmark suite that evaluates LLM responses using
real-world *user-experience* signals (thumbs-up / thumbs-down feedback collected
on a deployed conversational assistant), distilled into three complementary
tasks:

| Task | Name | What the model is asked to do | N |
|------|------|--------------------------------|--:|
| **1** | UX Judge      | Read a (user-turn, response) pair and decide whether the response is **Good** or **Bad** from a UX perspective | 2,000 |
| **2** | UX Eval       | Generate a response to a real user query that real users found hard to satisfy in the wild | 4,900 |
| **3** | UX Recovery   | Given a failed assistant response and a user-complaint follow-up turn, write a recovery response that the user would accept | 500 |

This repository accompanies the paper

> *"UXBench: A User-Experience Reward Benchmark for Foundation LLMs"* (under
> review, 2026).

It contains:

* **The benchmark itself** вЂ” query splits, public 200-sample eval demo, and
  full ground-truth labels for Task 1 (`src/uxbench-dataset/`).
* **All response files** generated by the **27 frontier LLMs** evaluated in the
  paper, for all three tasks (`src/experiments/task{1,2,3}_*/responses/`).
* **All GRM judge files** for Tasks 2 / 3 (`src/experiments/task{2,3}_*/judge/`).
* **All paper-cited ablations**, E1вЂ“E8 (`src/experiments/ablations/`).
* **Reusable code**: data pipeline (mining + auto-labeling), response
  generation harness, and the trained pointwise GRM judge
  (`src/data_pipeline/`, `src/response_generation/`, `src/grm_judge/`).

## At a glance

* **27 frontier LLMs evaluated** end-to-end on all three tasks
  (Gemini 2.5/3.x, GPT-5/5.x, Claude Opus 4.5/4.6/4.7 + Sonnet 4.5,
  DeepSeek R1 / V3 / V3.2 / V4 Pro, Doubao Seed 1.6 / 2.0 Lite/Pro,
  Hunyuan 3, GLM-5/5.1, Kimi K2.5/K2.6, MiniMax M2.5, Qwen3.6-Plus).
* **Unified CIDs** across every file:
  `uxbench_ux_judge_NNNN`, `uxbench_ux_eval_NNNN`, `uxbench_ux_recovery_NNNN`
  вЂ” so a row with a given new CID identifies the same underlying user turn in
  every model's response file, every judge file, and every ablation file.
* **No prompt / query leakage**: only AI-generated content (`generated_response`)
  is released; user queries, reasoning traces, and any internal context are
  stripped from every released file.

## Data schema

Every released `*.jsonl` file uses one JSON object per line. The released
schema is intentionally minimal:

### Task 1 вЂ” `task1_ux_judge/responses/<model>.jsonl`
```json
{
  "cid":          "uxbench_ux_judge_0001",
  "model":        "claude-opus-4.7",
  "ground_truth": 1,           // +1 = Good (thumbs-up),  -1 = Bad (thumbs-down)
  "difficulty":   "medium",
  "verdict":      1,           // model's predicted label (+1 / -1)
  "score":        0.92,
  "tokens":       412,
  "latency_s":    3.51
}
```

### Task 2 вЂ” `task2_ux_eval/responses/<model>.jsonl`
```json
{
  "cid":               "uxbench_ux_eval_0001",
  "model":             "gpt-5.5",
  "generated_response": "<the model's answer to the public query>",
  "tokens":            812,
  "latency_s":         5.93
}
```

### Task 3 вЂ” `task3_ux_recovery/responses/<model>.jsonl`
Same shape as Task 2 (`cid`, `model`, `generated_response`, `tokens`,
`latency_s`).

### GRM judge files вЂ” `task{2,3}_*/judge/judge_<model>.jsonl`
```json
{
  "cid":           "uxbench_ux_eval_0001",
  "judged_model":  "gpt-5.5",
  "verdict":       1,           // GRM's verdict   (+1 = Good, -1 = Bad)
  "score":         0.78,
  "good_logprob": -0.24,
  "bad_logprob":  -1.65,
  "grm_ok":        true,
  "latency_s":     0.42
}
```

## Reproducing the leaderboards

The three released `leaderboard.md` files (one per task) are computed directly
from the released `responses/` (Task 1) and `judge/` (Task 2/3) files. The
paper's main results table (27 models) reproduces exactly from these files.
